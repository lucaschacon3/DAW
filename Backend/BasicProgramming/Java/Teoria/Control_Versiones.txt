Sirven para llevar un registro de las versiones de un programa hecho normalmente en aquipo.
Los controles de versiones pueden ser:
    Centralizado.
    Distribuido -> Cada usuario tiene una parte.

Los repositarios pueden ser:
    - Locales -> Git.
    - Remotos -> GitHub o GitLab.

Iniciar sesion git:
    - git config --global user.name "Tu Nombre"
    - git config --global user.email "tu_correo@ejemplo.com"

Funciones git:
    - Clonar -> Clonar repositorio a local.
    - Commit -> Aplicar cambios.
    - Pull -> Adquirir la ultima version de Github en el IDE.
    - Push -> Subir codigo a Github.
    - Switch to -> Para cambiar de rama.
    - Merge -> Sincronizar ramas.
    - Staged Changes -> Cambios que quieres subir.

Ramas:
    - Hay ramas, la "branch master" (principal) y las que creamos nosotros.
    1º- Hay que pasar a tu rama mediante "switch To" y hacer "merge" con la Master para que en tu rama aparezca lo mismo, si no se sincroniza al momento hacer "pull".
    2º- Desde tu rama haces los cambios le das a "commit" y "push" para subirlo a GitHub.
    3º- Con "switch to" vamos a la Master y hacemos "merge" con la nuestra. Ya estarían sincronizadas. Solo queda hacer "commit" y "push". 
   
Pull Request:
    - La "rama Master" se puede proteger con "Pull Request " para que al hacer "merge" a la rama Master, el líder del equipo la revise, solucione conflictos y apruebe el codigo.
    - Al subirlo a la Master a ti te hacen Pull Request y cuando otro programador lo suba lo haces tú.
    - Hay que avisar a los participantes del equipo que la "rama master" esta actualizada.
